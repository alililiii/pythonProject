# 定向采购下单流程测试用例

> 页面入口：`https://ec-hwbeta.casstime.com/seller#/order/directional-purchase`  
> 测试账号：`ksseller1`  
> 源码范围：`icec-cloud-directional`  
> 生成日期：2026-05-12  

本文按“供应商为指定维修厂下定向采购单”的业务流程设计用例，不展开单个页面控件的细项校验。页面字段只作为流程输入，重点验证采购清单保存、结算、提交订单、客户审核支付以及异常场景下的数据一致性。

## 1. 测试范围

### 1.1 覆盖流程

1. 供应商登录定向采购页面。
2. 选择维修厂客户，确认收货地址、车辆品牌或通用件。
3. 选择发票需求。
4. 录入或批量粘贴商品，覆盖现货、调货、多商品、不同价格口径。
5. 点击下一步生成采购清单。
6. 提交给客户，客户审核并支付。
7. 后端保存定向采购主单和明细，结算后生成订单并回写购物车条目。

### 1.2 源码依据

| 模块/类 | 关键逻辑 | 测试关注点 |
|---|---|---|
| `DirectionalInfoServiceImpl.save` | 保存定向采购清单；校验请求；最多处理 200 条商品；保存前删除旧明细 | 清单生成、商品上限、二次提交覆盖旧明细 |
| `DirectionalServiceImpl.save` | 按买家、操作人、操作公司、店铺维度保证主单唯一 | 重复发起不创建重复主单 |
| `DirectionalItemServiceImpl.add/update` | 保存明细；结算后回写 `cartItemId` | 商品明细、购物车条目回写 |
| `DirectionalCommonBizServiceImpl.batchAddOrder` | 批量创建定向采购订单 | 批量下单、默认地址、默认仓、结算、提交订单 |
| `DirectionalCommonBizServiceImpl.directionalCheckout` | 组装立即结算数据，设置发票、仓库、售后政策、终端来源 | 结算数据正确性 |
| `DirectionalCommonBizServiceImpl.getSubmitOrderParamDTO` | 组装提交订单参数，设置 `channel=DIRECTIONAL`、物流仓库等 | 订单提交参数正确性 |

## 2. 测试数据准备

| 数据项 | 要求 |
|---|---|
| 供应商账号 | 使用 `ksseller1`，账号具备进入供应商工作台和定向采购页权限 |
| 维修厂客户 | 至少准备 2 个有效客户：一个有默认地址，一个无默认地址或地址不完整 |
| 收货地址 | 正常场景需完整到省、市、区/县、街道，并有收货人和手机号 |
| 车辆信息 | 准备普通车品牌场景和“无具体车品牌”通用件场景 |
| 发票需求 | 覆盖“需要发票”和“不需要发票” |
| 商品 | 准备现货商品、调货商品、轮胎商品、机油商品、禁售品牌商品、不存在商品、无发货仓商品 |
| 数量 | 覆盖 1 件、多件、200 条商品、超过 200 条商品 |
| 支付客户 | 客户侧账号可查看采购单并完成审核/支付，或测试环境可验证订单状态流转 |

## 9. 验收关注点

| 关注点 | 验收标准 |
|---|---|
| 采购单是否生成 | 供应商提交后，客户侧能看到对应采购清单 |
| 订单是否生成 | 客户审核并支付后，生成正式订单，订单来源为定向采购 |
| 主单唯一性 | 同一买家、操作人、操作公司、店铺维度不重复创建主单 |
| 明细一致性 | 页面商品、数量、价格、仓库、现货/调货与后端明细一致 |
| 价格口径 | 需要发票走含税价和含税钉箱费；不需要发票走不含税价和不含税钉箱费 |
| 商品上限 | 单次清单最多 200 条商品，超过上限必须有明确处理 |
| 异常提示 | 客户、地址、商品、仓库、结算、提交订单异常均有明确失败原因 |
| 数据回写 | 结算后 `cartItemId` 回写成功，可反查定向采购明细 |
